How It Works

Template to project
in one step.

The JUCE-Plugin-Starter template comes with an interactive shell script that asks 15+ questions one at a time. Miss one and you start over.

juce-dev replaces that with grouped prompts using Claude Code's AskUserQuestion UI. Your developer settings are loaded from the template's .env — if they look right, confirm once and move on. Want to change something? Walk through each setting individually.

All settings go to the new project only. The template stays untouched.

Capabilities

Everything the init script does, with a better interface and optional GPU UI integration.

  • Creates AU, VST3, and Standalone plugin projects
  • Auto-generates class names, namespaces, bundle IDs
  • Generates valid JUCE 4-letter plugin and manufacturer codes
  • Replaces all 9 template placeholders across source files
  • Optional Visage Metal GPU UI integration with patches
  • Optional DiagnosticKit for user crash reporting
  • Reuses developer settings from template .env
  • Initializes git and optionally creates GitHub repos
  • JUCE fetched via FetchContent with shared cache
  • Ready to build immediately after creation

Commands

Two commands.

Create a new project or add Visage to an existing one.

What you want Command What happens
Create a new plugin /juce-dev:create "My Synth" Copies template, replaces placeholders, generates .env, inits git. Optional: --visage for GPU UI, --no-github to skip repo creation
Add Visage GPU UI /juce-dev:setup-visage Adds Metal-accelerated Visage UI to an existing JUCE-Plugin-Starter project. Clones Visage, applies patches, replaces editor files

What Gets Generated

From name
to namespace.

All identifiers are derived automatically from the plugin name and developer name you provide.

# Input Plugin name: "My Cool Synth" Developer name: "Generous Corp" # Generated Class name: MyCoolSynth Project folder: my-cool-synth Namespace: generouscorp Bundle ID: com.generouscorp.my-cool-synth Plugin code: MYCO Manufacturer code: GECO
9 placeholders replaced across .cpp, .h, .cmake, .txt, .md, and .env files.

FAQ

Common questions.

What do I need installed before using this?

Required:

• Xcode with command-line tools
• CMake (brew install cmake)
• Claude Code

Optional:

• gh CLI (brew install gh) for GitHub repository creation
• Apple Developer account for code signing and distribution

JUCE is fetched automatically via CMake FetchContent and cached at ~/.juce_cache/. No manual JUCE installation needed.

Does this modify my template's .env?

No. The template's .env is read-only. Developer settings are loaded from it, but all changes are written only to the new project's .env. Your template stays untouched for the next project you create.

Will .env files get committed to git?

No. The JUCE-Plugin-Starter template includes .env in its .gitignore. Your developer credentials, Apple ID, and API keys are never committed or pushed to GitHub.

What is Visage?

Visage is a Metal-based GPU-accelerated UI framework for audio plugins on macOS. It provides rich widgets (text editors, popup menus, sliders), customizable themes, and 60fps rendering.

When you enable Visage, juce-dev clones the Visage repo, applies compatibility patches (keyboard handling, popup positioning, FPS caps, etc.), and replaces the standard JUCE editor with a Visage-powered one.

The juce-visage skill is available in Claude Code for ongoing Visage UI development guidance after project creation.

What are JUCE 4-letter codes?

JUCE requires two 4-character codes for plugin identification: a plugin code (from your plugin name) and a manufacturer code (from your developer name). These are used by DAWs like Logic Pro and Ableton to identify your plugin.

juce-dev generates these automatically using the same algorithm as the init script: for multi-word names it takes the first 2 characters of each word; for single words it picks strategic characters. The codes are always uppercase and exactly 4 characters.

Can I skip Apple Developer settings?

Yes. Apple Developer settings (Apple ID, Team ID, certificates, app-specific password) are only needed for code signing and distribution. You can skip all of them and add them later to your project's .env when you're ready to distribute.

You can build, test, and develop your plugin locally without any Apple Developer account.

How does this compare to the shell script?

The init_plugin_project.sh script asks 15+ questions one at a time in a terminal. If you make a mistake, you start over. It only works interactively.

juce-dev does the same thing but uses Claude Code's AskUserQuestion UI for grouped, visual prompts. If your developer settings are already configured, you can confirm them in one step instead of answering each question again. The underlying operations (rsync, sed, git init, gh repo create) are identical.
